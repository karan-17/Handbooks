1. Common Docker Issues and Solutions

Problem: "Cannot connect to the Docker daemon"
Solutions:
  1. Check if Docker daemon is running:
    2. sudo systemctl status docker
  2. Start Docker daemon if not running:
    3. sudo systemctl start docker
  3. Add user to docker group (avoid sudo):
    4. sudo usermod -aG docker $USER newgrp docker
  4. Check Docker socket permissions:
    5. ls -la /var/run/docker.sock
Problem: "Permission denied while trying to connect to Docker daemon socket"
Solutions:
  1. Check current user groups:
    2. groups $USER
  2. Add user to docker group:
    3. sudo usermod -aG docker $USER
  3. Log out and log back in, or run:
    4. newgrp docker
  4. Alternative temporary fix:
    5. sudo chmod 666 /var/run/docker.sock
Problem: "No space left on device"
Solutions:
  1. Check Docker disk usage:
    2. docker system df
  2. Clean up unused containers, networks, and images:
    3. docker system prune -a
  3. Remove specific unused resources:
    4. docker container prune docker image prune -a docker volume prune docker network prune
  4. Check available disk space:
    5. df -h

2. Container Startup and Runtime Problems

Problem: Container exits immediately after starting
Diagnostic Steps:
  1. Check container logs:
    2. docker logs <container_id>
  2. Inspect container configuration:
    3. docker inspect <container_id>
  3. Check exit code:
    4. docker ps -a
  4. Run container interactively for debugging:
    5. docker run -it <image> /bin/bash
Problem: Container runs but application doesn't respond
Solutions:
  1. Check if application is listening on correct port:
    2. docker exec <container_id> netstat -tlnp
  2. Verify port mapping:
    3. docker port <container_id>
  3. Check firewall rules:
    4. sudo ufw status sudo iptables -L
  4. Test connectivity from host:
    5. telnet localhost <port> curl -v localhost:<port>

Common Exit Codes

Exit Code Meaning Common Causes
0 Success Normal termination
1 General errors Application error, misconfiguration
125 Docker daemon error Container failed to run
126 Container command not executable Permission issues, wrong executable
127 Container command not found Path issues, missing binary
137 SIGKILL Out of memory, manual kill
143 SIGTERM Graceful termination

3. Image Build Failures and Debugging

Problem: "ADD failed: no source files were specified"
Solutions:
  1. Check file paths in Dockerfile:
    2. ls -la <source_file>
  2. Verify build context includes the files:
    3. docker build --no-cache -t <tag> .
  3. Check .dockerignore file:
    4. cat .dockerignore
  4. Use COPY instead of ADD for local files:
    5. COPY ./app /app
Problem: "Unable to find image" during build
Solutions:
  1. Check base image name and tag:
    2. docker search <image_name>
  2. Pull base image explicitly:
    3. docker pull <base_image>:<tag>
  3. Check registry connectivity:
    4. curl -I https://registry-1.docker.io/v2/
  4. Use specific registry URL if needed:
    5. FROM registry.hub.docker.com/library/ubuntu:20.04

Build Optimization Tips

Best Practices for Dockerfile

  • Use multi-stage builds to reduce image size
  • Order instructions by frequency of change (least to most)
  • Combine RUN commands to reduce layers
  • Use .dockerignore to exclude unnecessary files
  • Specify exact versions for base images

Debug Build Issues

# Build with detailed output docker build --progress=plain --no-cache -t myapp . # Build up to specific stage docker build --target <stage_name> -t debug_image . # Run intermediate container for debugging docker run -it <intermediate_image_id> /bin/sh

4. Network Connectivity Issues

Problem: Container cannot access external internet
Diagnostic Steps:
  1. Test DNS resolution inside container:
    2. docker exec <container> nslookup google.com
  2. Test internet connectivity:
    3. docker exec <container> ping 8.8.8.8
  3. Check Docker daemon DNS settings:
    4. docker info | grep -i dns
  4. Restart Docker daemon with custom DNS:
    5. sudo systemctl edit docker
    Add to override.conf:
    [Service]
    ExecStart=
    ExecStart=/usr/bin/dockerd --dns=8.8.8.8
Problem: Containers cannot communicate with each other
Solutions:
  1. Check if containers are on same network:
    2. docker network ls docker network inspect bridge
  2. Create custom network for container communication:
    3. docker network create mynetwork docker run --network mynetwork --name container1 <image1> docker run --network mynetwork --name container2 <image2>
  3. Test connectivity between containers:
    4. docker exec container1 ping container2
  4. Check iptables rules:
    5. sudo iptables -L DOCKER-USER

Network Diagnostic Commands

Network Inspection

# List networks docker network ls # Inspect network details docker network inspect <network_name> # Show container network settings docker inspect <container> | grep -A 10 NetworkSettings

Port and Connectivity

# Check port mappings docker port <container> # Test port connectivity telnet <host> <port> nc -zv <host> <port> # Show listening ports in container docker exec <container> netstat -tlnp

5. Volume and Storage Problems

Problem: Volume mount not working or files not persisting
Solutions:
  1. Check volume mount syntax:
    2. docker run -v /host/path:/container/path <image>
  2. Verify host directory exists and has correct permissions:
    3. ls -la /host/path sudo chown -R $USER:$USER /host/path
  3. Check SELinux context (if applicable):
    4. ls -Z /host/path chcon -Rt svirt_sandbox_file_t /host/path
  4. Use named volumes instead of bind mounts:
    5. docker volume create myvolume docker run -v myvolume:/container/path <image>
Problem: "No space left on device" for volumes
Solutions:
  1. Check Docker volume usage:
    2. docker system df -v
  2. List and inspect volumes:
    3. docker volume ls docker volume inspect <volume_name>
  3. Remove unused volumes:
    4. docker volume prune
  4. Change Docker root directory:
    5. sudo systemctl stop docker sudo mv /var/lib/docker /new/path/docker sudo ln -s /new/path/docker /var/lib/docker

Volume Management Commands

# Create named volume docker volume create myvolume # List volumes with details docker volume ls --format "table {{.Name}}\t{{.Driver}}\t{{.Scope}}" # Inspect volume docker volume inspect myvolume # Remove unused volumes docker volume prune # Backup volume data docker run --rm -v myvolume:/data -v $(pwd):/backup ubuntu tar czf /backup/backup.tar.gz /data # Restore volume data docker run --rm -v myvolume:/data -v $(pwd):/backup ubuntu tar xzf /backup/backup.tar.gz -C /

  3. Check I/O performance:

7. Memory and Resource Constraints

Problem: Container killed due to out of memory (OOMKilled)
Solutions:
  1. Check container memory limits:
    2. docker inspect <container> | grep -i memory
  2. Monitor memory usage:
    3. docker stats <container> docker exec <container> cat /proc/meminfo
  3. Increase memory limit:
    4. docker run --memory=2g <image>
  4. Check system memory:
    5. free -h dmesg | grep -i "killed process"
Problem: High CPU usage by container
Solutions:
  1. Check CPU usage:
    2. docker stats <container>
  2. Limit CPU usage:
    3. docker run --cpus="1.5" <image> docker run --cpu-shares=512 <image>
  3. Analyze CPU-intensive processes:
    4. docker exec <container> top -p 1
  4. Check for CPU throttling:
    5. docker exec <container> cat /sys/fs/cgroup/cpu/cpu.stat

Resource Limits Configuration

Resource Docker Flag Example Description
Memory --memory --memory=1g Maximum memory usage
CPU --cpus --cpus="1.5" Number of CPUs
CPU Shares --cpu-shares --cpu-shares=512 Relative CPU weight
Swap --memory-swap --memory-swap=2g Memory + swap limit

8. Log Analysis and Debugging Techniques

Container Logging Commands

# View container logs docker logs <container> # Follow logs in real-time docker logs -f <container> # Show last N lines docker logs --tail 50 <container> # Show logs since specific time docker logs --since "2023-01-01T00:00:00" <container> # Show logs with timestamps docker logs -t <container> # Export logs to file docker logs <container> > container.log 2>&1
Problem: Container logs are not appearing
Solutions:
  1. Check if application writes to stdout/stderr:
    2. docker exec <container> ls -la /proc/1/fd/
  2. Configure logging driver:
    3. docker run --log-driver=json-file <image>
  3. Check Docker daemon logging configuration:
    4. docker info | grep -i logging
  4. Use log aggregation tools:
    5. docker run --log-driver=syslog <image>

Log Analysis Tools

Built-in Tools

# Grep for errors docker logs <container> | grep -i error # Count log lines docker logs <container> | wc -l # Parse JSON logs docker logs <container> | jq '.'

External Tools

# Using ELK stack docker run -d --name elasticsearch elasticsearch:7.9.0 docker run -d --name logstash logstash:7.9.0 docker run -d --name kibana kibana:7.9.0 # Using Fluentd docker run --log-driver=fluentd <image>

9. Docker Daemon Issues

Problem: Docker daemon won't start
Diagnostic Steps:
  1. Check daemon status:
    2. sudo systemctl status docker
  2. Check daemon logs:
    3. sudo journalctl -u docker.service -f
  3. Check for conflicting processes:
    4. ps aux | grep docker
  4. Check daemon configuration:
    5. cat /etc/docker/daemon.json
  5. Start daemon manually for debugging:
    6. sudo dockerd --debug
Problem: Docker daemon consuming high resources
Solutions:
  1. Monitor daemon resource usage:
    2. ps aux | grep dockerd top -p $(pidof dockerd)
  2. Check for runaway containers:
    3. docker stats --all
  3. Clean up Docker system:
    4. docker system prune -a
  4. Restart Docker daemon:
    5. sudo systemctl restart docker

Daemon Configuration

Common daemon.json settings:

{ "log-driver": "json-file", "log-opts": { "max-size": "10m", "max-file": "3" }, "storage-driver": "overlay2", "dns": ["8.8.8.8", "8.8.4.4"], "insecure-registries": ["myregistry:5000"] }

10. Platform-Specific Problems

Windows
macOS

Windows Issues

Problem: "Drive has not been shared" on Windows
Solutions:
  1. Enable drive sharing in Docker Desktop settings
  2. Reset credentials in Docker Desktop
  3. Check Windows credentials manager
  4. Use UNC paths for network drives:
    5. docker run -v //server/share:/data <image>
Problem: Windows container cannot access Windows features
Solutions:
  1. Use Windows base images:
    2. FROM mcr.microsoft.com/windows/servercore:ltsc2019
  2. Enable Windows containers mode in Docker Desktop
  3. Check Windows version compatibility

macOS Issues

Problem: File sharing performance issues on macOS
Solutions:
  1. Use delegated or cached mount options:
    2. docker run -v /host/path:/container/path:delegated <image>
  2. Enable file sharing for specific directories in Docker Desktop
  3. Use named volumes instead of bind mounts for better performance
  4. Consider using Docker volume plugins
Problem: Docker Desktop crashes on macOS
Solutions:
  1. Check macOS compatibility
  2. Reset Docker Desktop to factory defaults
  3. Check system logs:
    4. log show --predicate 'process == "Docker"' --last 1h
  4. Allocate more resources in Docker Desktop settings

Linux Issues

Problem: SELinux preventing container operations
Solutions:
  1. Check SELinux status:
    2. sestatus
  2. Use Z or z flags for volume mounts:
    3. docker run -v /host/path:/container/path:Z <image>
  3. Set SELinux context:
    4. chcon -Rt svirt_sandbox_file_t /host/path
  4. Check SELinux logs:
    5. ausearch -m avc -ts recent
Problem: AppArmor blocking container operations
Solutions:
  1. Check AppArmor status:
    2. aa-status
  2. Disable AppArmor for Docker:
    3. docker run --security-opt apparmor=unconfined <image>
  3. Check AppArmor logs:
    4. dmesg | grep apparmor

11. Docker Compose Troubleshooting

Problem: "Service 'web' failed to build" in Docker Compose
Solutions:
  1. Check compose file syntax:
    2. docker-compose config
  2. Build with verbose output:
    3. docker-compose build --progress=plain --no-cache
  3. Check build context and Dockerfile path:
    4. ls -la ./path/to/Dockerfile
  4. Validate compose file version compatibility
Problem: Services cannot communicate in Docker Compose
Solutions:
  1. Check service names and network configuration:
    2. docker-compose ps docker network ls
  2. Use service names for inter-service communication:
    3. # Use 'db' instead of 'localhost' DATABASE_URL=postgresql://user:pass@db:5432/mydb
  3. Check port exposure vs publishing:
    4. # Expose port to other services only expose: - "5432" # Publish port to host ports: - "5432:5432"

Docker Compose Debugging Commands

# Validate compose file docker-compose config # Check service logs docker-compose logs <service_name> # Check service status docker-compose ps # Restart specific service docker-compose restart <service_name> # Force recreate containers docker-compose up --force-recreate # Run one-off commands docker-compose exec <service> /bin/bash # Check networks created by compose docker network ls | grep <project_name>

12. Security and Permission Issues

Problem: "Permission denied" when accessing files in container
Solutions:
  1. Check file ownership and permissions:
    2. ls -la /host/path docker exec <container> ls -la /container/path
  2. Fix ownership on host:
    3. sudo chown -R $USER:$USER /host/path
  3. Use user namespace mapping:
    4. docker run --user $(id -u):$(id -g) <image>
  4. Set proper permissions in Dockerfile:
    5. RUN chown -R appuser:appuser /app USER appuser
Problem: Root privilege escalation concerns
Solutions:
  1. Run containers as non-root user:
    2. docker run --user 1000:1000 <image>
  2. Use security options:
    3. docker run --security-opt no-new-privileges <image>
  3. Drop capabilities:
    4. docker run --cap-drop ALL --cap-add NET_ADMIN <image>
  4. Use read-only containers:
    5. docker run --read-only <image>

Security Best Practices

Container Security Checklist

  • Run containers as non-root users
  • Use minimal base images (Alpine, distroless)
  • Regularly update base images and dependencies
  • Scan images for vulnerabilities
  • Use secrets management for sensitive data
  • Implement resource limits
  • Use read-only filesystems when possible
  • Enable Docker Content Trust

13. Registry and Authentication Problems

Problem: "Authentication required" when pulling images
Solutions:
  1. Login to registry:
    2. docker login <registry_url>
  2. Check stored credentials:
    3. cat ~/.docker/config.json
  3. Use credential helpers:
    4. # For AWS ECR docker login --username AWS --password-stdin <ecr_uri>
  4. Set registry in image name:
    5. docker pull registry.example.com/myimage:latest
Problem: "X509 certificate signed by unknown authority"
Solutions:
  1. Add registry to insecure registries:
    2. # In /etc/docker/daemon.json { "insecure-registries": ["myregistry:5000"] }
  2. Add CA certificate:
    3. sudo mkdir -p /etc/docker/certs.d/myregistry:5000 sudo cp ca.crt /etc/docker/certs.d/myregistry:5000/
  3. Skip TLS verification (not recommended for production):
    4. docker run --insecure-registry myregistry:5000 <image>

15. Prevention Best Practices

Development Best Practices

  • Use multi-stage builds to reduce final image size
  • Pin base image versions for reproducible builds
  • Leverage build cache by ordering Dockerfile instructions properly
  • Use .dockerignore to exclude unnecessary files
  • Run containers as non-root for security
  • Set resource limits to prevent resource exhaustion
  • Use health checks for container monitoring
  • Implement proper logging to stdout/stderr

Production Best Practices

  • Use orchestration platforms (Kubernetes, Docker Swarm)
  • Implement monitoring and alerting
  • Regular security scanning of images
  • Backup strategy for volumes and data
  • Network segmentation for security
  • Secrets management for sensitive data
  • Regular updates of Docker and base images
  • Disaster recovery planning

Preventive Maintenance Commands

# Regular cleanup (run weekly) docker system prune -f # Deep cleanup (run monthly) docker system prune -a -f --volumes # Update all images (with caution) docker images --format "table {{.Repository}}:{{.Tag}}" | grep -v REPOSITORY | xargs -I {} docker pull {} # Check for security vulnerabilities docker scan <image_name> # Monitor disk usage watch -n 30 'docker system df' # Backup important volumes docker run --rm -v volume_name:/data -v $(pwd):/backup ubuntu tar czf /backup/backup-$(date +%Y%m%d).tar.gz /data

Troubleshooting Checklist

When encountering Docker issues, follow this checklist:

  1. Check Docker daemon status and logs
  2. Verify container logs for application errors
  3. Inspect container and image configurations
  4. Test network connectivity and port mappings
  5. Monitor resource usage (CPU, memory, disk)
  6. Verify volume mounts and permissions
  7. Check for conflicting containers or ports
  8. Review security settings and constraints
  9. Test with minimal example to isolate the issue
  10. Consult Docker documentation and community resources
Important: Always test troubleshooting steps in a development environment before applying them to production systems. Keep backups of important data and configurations.
