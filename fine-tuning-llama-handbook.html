Fine-Tuning Fundamentals

What is Fine-Tuning?

Fine-tuning is the process of taking a pre-trained model and adapting it to a specific task or domain by training it on a smaller, specialized dataset. This approach leverages the knowledge already learned during pre-training while specializing the model for your particular use case.

Key Benefits of Fine-Tuning

  • Efficiency: Requires significantly less data and computation than training from scratch
  • Performance: Achieves 80-95% of custom model quality with fraction of the resources
  • Speed: Training completed in hours instead of days or weeks
  • Accessibility: Modern techniques enable fine-tuning on consumer hardware

Fine-Tuning vs Other Training Approaches

Approach Data Requirements Compute Needs Time to Train Best Use Case
Pre-training Trillions of tokens Thousands of GPUs Weeks/Months Building foundation models
Fine-tuning 500-10,000+ examples 1-8 GPUs Hours/Days Domain specialization
Transfer Learning 100-1,000 examples Single GPU Minutes/Hours Similar domain adaptation
In-Context Learning Few examples Inference only Immediate Quick prototyping

When to Use Fine-Tuning

✅ Use Fine-Tuning When

  • You have 500+ high-quality examples
  • Need consistent specialized behavior
  • Domain expertise beyond general knowledge
  • Quality requirements justify compute costs

❌ Avoid Fine-Tuning When

  • Dataset is too small (<100 examples)
  • Task can be solved with prompting
  • Limited computational resources
  • Need frequent model updates

Types of Fine-Tuning

Full Fine-Tuning vs Parameter-Efficient Fine-Tuning (PEFT)

1

Full Fine-Tuning

Updates all model parameters. Provides maximum quality but requires enormous memory (16 bytes per parameter).

2

Parameter-Efficient Fine-Tuning (PEFT)

Updates only a small subset of parameters (0.1-3%). Reduces memory by 10-20x while maintaining 80-95% quality.

LoRA & QLoRA Deep Dive

LoRA (Low-Rank Adaptation)

LoRA decomposes weight updates into low-rank matrices, dramatically reducing the number of trainable parameters while maintaining model quality.

LoRA Mathematical Foundation

Instead of updating the full weight matrix W, LoRA approximates the update as:

W' = W + α(BA)

Where B and A are low-rank matrices (r << d), and α is a scaling factor.

Key LoRA Parameters

Rank (r)

Controls the number of trainable parameters. Higher rank = more parameters but better adaptation capability. Typical values: 8, 16, 32, 64.

Alpha (α)

Scaling factor for LoRA updates. Usually set to rank value or slightly higher. Controls how much the adaptation affects the model.

Dropout

Regularization to prevent overfitting. Typical values: 0.05-0.1 for small datasets, 0.1-0.2 for larger datasets.

Target Modules

Which layers to apply LoRA to. Common choices: query/value projections, all linear layers, or attention layers only.

QLoRA (Quantized LoRA)

QLoRA combines LoRA with 4-bit quantization, enabling fine-tuning of massive models on consumer hardware.

QLoRA Innovations

  • 4-bit Quantization: Reduces base model memory by 75%
  • Double Quantization: Quantizes quantization constants for additional savings
  • Paged Optimizers: Handles memory spikes by paging to CPU memory
  • NormalFloat4 (NF4): Information-theoretically optimal for normally distributed weights

Fine-Tuning Techniques

Data Preparation Best Practices

1

Dataset Collection

Gather high-quality, diverse examples. Quality matters more than quantity - 1,000 good examples beat 10,000 poor ones.

2

Data Cleaning

Remove noise, duplicates, and irrelevant content. Ensure consistency in formatting and style.

3

Format Standardization

Convert data to consistent format (instruction-response, conversational, or completion style).

4

Train-Validation Split

Reserve 10-20% for validation. Ensure splits maintain distribution balance.

Hyperparameter Optimization

Parameter Typical Range Impact Tuning Strategy
Learning Rate 1e-5 to 5e-4 Training stability and speed Start with 2e-4, reduce if unstable
Batch Size 1-16 Memory usage and convergence Max that fits in memory
LoRA Rank 8-128 Model capacity vs efficiency Start with 64, adjust based on complexity
LoRA Alpha 16-128 Adaptation strength Usually 1x to 2x the rank

Quantization Setup

Cell 5: Quantization Configuration

# Configure 4-bit quantization for memory efficiency
quant_config = BitsAndBytesConfig(
    load_in_4bit=True,                      # Enable 4-bit loading
    bnb_4bit_quant_type="nf4",              # Use NormalFloat4
    bnb_4bit_compute_dtype=torch.float16,   # Compute in FP16
    bnb_4bit_use_double_quant=False,        # Disable nested quantization
)

# Benefits of this configuration:
# - Reduces memory usage by ~75%
# - NF4 is optimal for normally distributed weights
# - FP16 compute maintains training stability

Cell 6: Model Loading with Quantization

# Load base model with quantization applied
model = AutoModelForCausalLM.from_pretrained(
    base_model,                          # Model identifier
    quantization_config=quant_config,    # Apply quantization
    device_map={"": 0}                   # Use first GPU
)

# Disable gradient computation for base model
model.config.use_cache = False
model.config.pretraining_tp = 1

print(f"Model loaded on device: {next(model.parameters()).device}")

Cell 7: Tokenizer Setup

# Load and configure tokenizer
tokenizer = AutoTokenizer.from_pretrained(base_model, trust_remote_code=True)

# Configure padding token (required for batch training)
tokenizer.pad_token = tokenizer.eos_token
tokenizer.padding_side = "right"

# Tokenizer configuration importance:
# - Pad token enables batch processing
# - Right padding is standard for causal LM
# - Ensures proper sequence handling

⚠️ Important: Tokenizer Padding

Setting the pad_token is crucial for batch training. Without it, the trainer cannot properly batch sequences of different lengths, leading to training errors.

Training Process

Cell 8: LoRA Configuration

# Configure LoRA for parameter-efficient fine-tuning
peft_args = LoraConfig(
    lora_alpha=16,              # Scaling factor for LoRA updates
    lora_dropout=0.1,           # Dropout for regularization
    r=64,                      # Rank of LoRA matrices
    bias="none",               # Don't adapt bias terms
    task_type="CAUSAL_LM",      # Language modeling task
)

# Parameter explanation:
# - r=64: Creates ~0.6% trainable parameters
# - alpha=16: Moderate scaling for stable training
# - dropout=0.1: Prevents overfitting on small dataset

Cell 9: Training Arguments

# Configure training hyperparameters
training_params = TrainingArguments(
    output_dir="./results",              # Output directory
    num_train_epochs=1,               # Number of epochs
    per_device_train_batch_size=4,    # Batch size per GPU
    gradient_accumulation_steps=1,    # Gradient accumulation
    optim="paged_adamw_32bit",         # Memory-efficient optimizer
    save_steps=25,                   # Checkpoint frequency
    logging_steps=25,               # Logging frequency
    learning_rate=2e-4,             # Learning rate
    weight_decay=0.001,             # L2 regularization
    fp16=False,                     # Mixed precision
    bf16=False,                     # BFloat16 precision
    max_grad_norm=0.3,              # Gradient clipping
    max_steps=-1,                   # Use epochs instead
    warmup_ratio=0.03,              # Learning rate warmup
    group_by_length=True,           # Group similar lengths
    lr_scheduler_type="constant",     # Constant learning rate
    report_to="tensorboard"           # Logging backend
)

Cell 10: Trainer Initialization

# Initialize the SFT (Supervised Fine-Tuning) trainer
trainer = SFTTrainer(
    model=model,                         # Quantized model
    train_dataset=dataset,               # Training data
    peft_config=peft_args,               # LoRA configuration
    dataset_text_field="text",          # Text column name
    max_seq_length=None,              # Auto-detect max length
    tokenizer=tokenizer,                 # Configured tokenizer
    args=training_params,                # Training arguments
    packing=False,                    # Don't pack sequences
)

# SFTTrainer handles:
# - LoRA adapter initialization
# - Gradient setup for trainable parameters
# - Data loading and batching

Cell 11: Training Execution

# Start the training process
trainer.train()

# Training will:
# 1. Process 1000 examples in batches of 4
# 2. Update only LoRA parameters (~4M out of 7B)
# 3. Save checkpoints every 25 steps
# 4. Log metrics to TensorBoard
# 5. Complete 250 training steps (1000/4 = 250)

Advanced Topics

Memory Optimization Techniques

Gradient Checkpointing

Trade computation for memory by recomputing activations during backprop. Enables training larger models with limited VRAM.

Gradient Accumulation

Simulate larger batch sizes by accumulating gradients over multiple forward passes before updating.

Mixed Precision

Use FP16/BF16 for forward pass and FP32 for gradient computation. Reduces memory by ~50%.

CPU Offloading

Move optimizer states and inactive parameters to CPU memory, enabling training of larger models.

Troubleshooting

Common Issues and Solutions

CUDA Out of Memory

  • Reduce batch size to 1
  • Enable gradient checkpointing
  • Use gradient accumulation
  • Switch to QLoRA if using LoRA

Tokenizer Warnings

  • Set tokenizer.pad_token = tokenizer.eos_token
  • Configure padding_side = "right"
  • Update transformers library
  • Check trust_remote_code=True

Slow Training Speed

  • Enable mixed precision (fp16/bf16)
  • Use compiled model (torch.compile)
  • Optimize batch size for your GPU
  • Enable group_by_length

Model Quality Issues

  • Check dataset quality and formatting
  • Increase LoRA rank (r parameter)
  • Adjust learning rate
  • Add more training epochs

Best Practices

Data Quality Guidelines

📊 Data Quality Checklist

  • Diversity: Include varied examples covering different scenarios
  • Quality over Quantity: 1,000 high-quality examples > 10,000 poor ones
  • Consistent Formatting: Standardize instruction-response format
  • Length Distribution: Mix of short and long examples
  • Domain Relevance: Ensure examples match your use case

Training Best Practices

1

Start Conservative

Begin with lower learning rates (1e-5) and increase if training is stable.

2

Monitor Overfitting

Use validation sets and early stopping to prevent overfitting on small datasets.

3

Experiment Incrementally

Change one parameter at a time to understand impact on model performance.

4

Save Checkpoints

Regular checkpointing enables recovery from failures and comparison of different stages.

Production Deployment Tips

  • Model Merging: Merge LoRA weights for faster inference
  • Quantization: Apply post-training quantization for deployment
  • Batch Processing: Group similar-length sequences for efficiency
  • Caching: Implement KV-cache for faster autoregressive generation
  • Monitoring: Track inference latency, throughput, and quality metrics
