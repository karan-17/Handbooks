🐳 What is Docker?

Definition and Core Concept

Docker is a containerization platform that enables developers to package applications and their dependencies into lightweight, portable containers. These containers can run consistently across different environments, from development laptops to production servers.

Key Principle: "Build once, run anywhere" - Docker ensures that your application runs the same way regardless of the underlying infrastructure.

History and Evolution

  • 2013: Docker was founded by Solomon Hykes and launched as an open-source project
  • 2014: Docker 1.0 released with production-ready container runtime
  • 2016: Docker Swarm mode introduced for container orchestration
  • 2017: Kubernetes integration and enterprise focus
  • 2019: Docker Desktop launched for developers
  • 2021-2024: Focus on developer experience and security
  • 2025-2026: AI-assisted development and WebAssembly integration

What Makes Docker Special?

  • Lightweight: Containers share the host OS kernel, making them much smaller than VMs
  • Fast: Start containers in milliseconds vs. minutes for VMs
  • Portable: Consistent behavior across development, staging, and production
  • Scalable: Easy to scale applications up or down
  • Efficient: Better resource utilization compared to virtual machines

Docker Engine

Docker Engine is the core runtime that manages containers on your system. It consists of:

Docker Daemon (dockerd)

  • Long-running background service
  • Manages Docker objects (images, containers, networks, volumes)
  • Listens for Docker API requests
  • Handles container lifecycle management
  • Communicates with other Docker daemons

Docker Client

  • Command-line interface (CLI)
  • Sends commands to Docker daemon via REST API
  • Can connect to local or remote Docker daemons
  • Primary way users interact with Docker

Docker REST API

  • Communication interface between client and daemon
  • Can be accessed over UNIX sockets or network interface
  • Enables programmatic interaction with Docker
  • Foundation for third-party tools and integrations

🔑 Key Docker Concepts

Images

Docker images are read-only templates used to create containers. They contain:

  • Application code and dependencies
  • Runtime environment
  • System libraries
  • Configuration files
  • Environment variables
# Image management commands # List all images docker images # Build an image from Dockerfile docker build -t myapp:1.0 . # Remove an image docker rmi myapp:1.0 # View image history docker history nginx

Containers

Containers are running instances of Docker images. They are:

  • Isolated processes with their own filesystem
  • Portable and consistent across environments
  • Lightweight and fast to start/stop
  • Stateless by design (data persists in volumes)
# Container management commands # Run a container docker run -d --name webserver -p 80:80 nginx # List running containers docker ps # Stop a container docker stop webserver # Remove a container docker rm webserver # Execute command in running container docker exec -it webserver bash

Layers

Docker images are built using a layered filesystem:

Image Layer Structure

Application Layer

Dependencies Layer

Runtime Layer

OS Layer (Base Image)
  • Layer Sharing: Common layers are shared between images
  • Caching: Unchanged layers are cached for faster builds
  • Efficiency: Only changed layers need to be downloaded/uploaded
  • Version Control: Each layer represents a change in the image

Registries

Public Registries

Private Registries

Dockerfile

Text file with instructions to build Docker images:

# Example Dockerfile for a Node.js application FROM node:18-alpine # Set working directory WORKDIR /app # Copy package files COPY package*.json ./ # Install dependencies RUN npm ci --only=production # Copy application code COPY . . # Expose port EXPOSE 3000 # Define the command to run the application CMD ["npm", "start"]

Volumes and Networking

Volumes

  • Purpose: Persistent data storage outside containers
  • Types: Named volumes, bind mounts, tmpfs mounts
  • Benefits: Data persistence, sharing between containers
# Volume examples # Create a named volume docker volume create mydata # Run container with volume docker run -v mydata:/app/data myapp # Bind mount host directory docker run -v /host/path:/container/path myapp

Networking

  • Bridge Network: Default isolated network
  • Host Network: Use host's network stack
  • Custom Networks: User-defined networks for container communication

Step-by-Step Getting Started

Step 1: Install Docker
# Linux (Ubuntu/Debian) curl -fsSL https://get.docker.com -o get-docker.sh sh get-docker.sh # Add user to docker group sudo usermod -aG docker $USER # Verify installation docker --version docker run hello-world
Step 2: Create Your First Application
# Create a simple web application mkdir my-docker-app cd my-docker-app # Create app.js cat > app.js << 'EOF' const express = require('express'); const app = express(); const port = 3000; app.get('/', (req, res) => { res.send('Hello Docker World!'); }); app.listen(port, () => { console.log(`App listening at http://localhost:${port}`); }); EOF # Create package.json cat > package.json << 'EOF' { "name": "my-docker-app", "version": "1.0.0", "main": "app.js", "dependencies": { "express": "^4.18.2" } } EOF
Step 3: Create a Dockerfile
# Create Dockerfile cat > Dockerfile << 'EOF' FROM node:18-alpine WORKDIR /app COPY package*.json ./ RUN npm install COPY . . EXPOSE 3000 CMD ["node", "app.js"] EOF
Step 4: Build and Run
# Build the image docker build -t my-docker-app . # Run the container docker run -p 3000:3000 my-docker-app # Test the application curl http://localhost:3000 # Output: Hello Docker World!
Step 5: Essential Commands
# Container management docker ps # List running containers docker ps -a # List all containers docker stop container_name # Stop a container docker rm container_name # Remove a container # Image management docker images # List images docker rmi image_name # Remove an image docker pull ubuntu:latest # Pull an image # Debugging docker logs container_name # View container logs docker exec -it container_name bash # Enter container

Best Practices for Beginners

  • Use .dockerignore: Exclude unnecessary files from build context
  • Multi-stage builds: Optimize image sizes
  • Non-root users: Run containers as non-root for security
  • Health checks: Monitor container health
  • Environment variables: Externalize configuration
  • Volume management: Persist important data
# Example .dockerignore node_modules npm-debug.log .git .gitignore README.md .env coverage .nyc_output # Example multi-stage Dockerfile # Build stage FROM node:18-alpine AS builder WORKDIR /app COPY package*.json ./ RUN npm ci --only=production # Production stage FROM node:18-alpine RUN addgroup -g 1001 -S nodejs RUN adduser -S nextjs -u 1001 WORKDIR /app COPY --from=builder /app/node_modules ./node_modules COPY . . USER nextjs CMD ["node", "app.js"]

Common Beginner Pitfalls to Avoid

  • Large images: Use alpine or slim base images
  • Root privileges: Always run as non-root user
  • Secrets in images: Never include passwords or keys
  • Data loss: Use volumes for persistent data
  • Network issues: Understand port mapping and networking

Image Optimization Strategies

  • Multi-stage builds: Separate build and runtime environments
  • Layer caching: Order Dockerfile instructions efficiently
  • Base image selection: Choose minimal base images
  • Dependency management: Install only production dependencies
  • Static analysis: Remove unused packages and files
# Optimized multi-stage Dockerfile # Build stage FROM node:18-alpine AS dependencies WORKDIR /app COPY package*.json ./ RUN npm ci --only=production && npm cache clean --force # Build stage FROM node:18-alpine AS build WORKDIR /app COPY package*.json ./ RUN npm ci COPY . . RUN npm run build # Runtime stage FROM node:18-alpine AS runtime RUN addgroup -g 1001 -S nodejs RUN adduser -S nextjs -u 1001 WORKDIR /app COPY --from=dependencies /app/node_modules ./node_modules COPY --from=build /app/dist ./dist COPY --from=build /app/package.json ./package.json USER nextjs EXPOSE 3000 CMD ["npm", "start"]

⚙️ Container Orchestration Introduction

Why Orchestration?

As applications grow beyond single containers, orchestration becomes essential for:

  • Service Discovery: How containers find and communicate with each other
  • Load Balancing: Distributing traffic across container instances
  • Health Management: Monitoring and replacing unhealthy containers
  • Scaling: Adding or removing containers based on demand
  • Rolling Updates: Deploying new versions without downtime
  • Resource Management: Optimizing resource allocation across clusters

Kubernetes Introduction

The industry-standard container orchestration platform:

Core Concepts

Pods

  • Smallest deployable units
  • Group of one or more containers
  • Shared network and storage
  • Ephemeral by nature

Services

  • Stable network endpoints
  • Load balancing across pods
  • Service discovery mechanism
  • External access management

Deployments

  • Desired state management
  • Rolling update strategies
  • Rollback capabilities
  • Replica management

Choosing the Right Orchestration

When to Use Docker Compose

  • Local development environments
  • Simple multi-container applications
  • Quick prototyping and testing
  • Single-host deployments

When to Use Kubernetes

  • Production workloads requiring high availability
  • Applications needing auto-scaling
  • Complex microservices architectures
  • Multi-cloud or hybrid cloud deployments
  • Enterprise environments with compliance requirements

Migration Path

Step 1: Start with Docker Compose for Development
Step 2: Learn Container Fundamentals
Step 3: Explore Simple Orchestration (Swarm)
Step 4: Graduate to Kubernetes for Production

Orchestration Best Practices

  • Start Simple: Begin with Docker Compose for development
  • Plan for Scale: Design applications with orchestration in mind
  • Health Checks: Implement proper health monitoring
  • Graceful Shutdown: Handle termination signals properly
  • Resource Limits: Set appropriate CPU and memory limits
  • Security: Follow security best practices for container orchestration
2026 Recommendation: Start with Docker Compose for development, use managed Kubernetes services (EKS, GKE, AKS) for production. The hybrid approach of containers in VMs provides the best balance of performance, security, and operational simplicity.
