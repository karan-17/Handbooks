AWS EKS Overview

Amazon Elastic Kubernetes Service (EKS) is a fully managed Kubernetes service that makes it easy to run Kubernetes on AWS without needing to install and operate your own Kubernetes control plane.

Key Benefits

  • Fully Managed: AWS manages the Kubernetes control plane
  • High Availability: Multi-AZ control plane deployment
  • Security: Integrated with AWS IAM, VPC, and security groups
  • Scalability: Auto-scaling worker nodes and pods
  • Integration: Native integration with AWS services (ALB, EBS, EFS, ECR)
  • Compliance: SOC, PCI, ISO, FedRAMP certifications
  • Cost-Effective: Pay only for worker nodes, control plane is managed
EKS vs Self-Managed:
With EKS, you focus on your applications while AWS handles the complexity of running, updating, and securing the Kubernetes control plane.

EKS Pricing

Component Cost Notes
EKS Control Plane $0.10/hour per cluster ~$72/month regardless of usage
Worker Nodes (EC2) Standard EC2 pricing t3.medium ~$30/month
Fargate Pods Per vCPU and memory More expensive but serverless
Load Balancers ALB: $16-22/month For ingress traffic
EBS Volumes $0.10/GB/month (gp2) For persistent storage

# Install AWS CLI (macOS)
brew install awscli

curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"

curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"

curl --silent --location "https://github.com/weaveworks/eksctl/releases/latest/download/eksctl_$(uname -s)_amd64.tar.gz" | tar xz -C /tmp
sudo mv /tmp/eksctl /usr/local/bin

# Verify installations
aws --version
kubectl version --client
eksctl version

Required IAM Permissions

Your AWS user/role needs the following permissions:
  • AmazonEKSClusterPolicy
  • AmazonEKSWorkerNodePolicy
  • AmazonEKS_CNI_Policy
  • AmazonEC2ContainerRegistryReadOnly
  • EC2 permissions for VPC/subnet management
  • CloudFormation permissions (for eksctl)

EKS Cluster Setup

Method 1: Using eksctl (Recommended)

eksctl is the easiest way to create and manage EKS clusters.

# Create a basic EKS cluster
eksctl create cluster \
  --name my-cluster \
  --version 1.28 \
  --region us-west-2 \
  --nodegroup-name standard-workers \
  --node-type t3.medium \
  --nodes 3 \
  --nodes-min 1 \
  --nodes-max 4 \
  --managed

# This command will:
# 1. Create a VPC with public and private subnets
# 2. Create an EKS cluster with the control plane
# 3. Create a managed node group with 3 EC2 instances
# 4. Configure kubectl to connect to the cluster
# 5. Takes about 15-20 minutes to complete
Cost Alert: This will create billable AWS resources. The cluster costs ~$72/month plus EC2 instance costs.

Advanced eksctl Configuration

# Create cluster with custom configuration
cat << EOF > cluster-config.yaml
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: production-cluster
  region: us-west-2
  version: "1.28"

availabilityZones: ["us-west-2a", "us-west-2b", "us-west-2c"]

managedNodeGroups:
  - name: general-workers
    instanceType: t3.medium
    amiFamily: AmazonLinux2
    minSize: 2
    maxSize: 10
    desiredCapacity: 3
    volumeSize: 20
    ssh:
      enable: true
    tags:
      Environment: production
      Team: platform

  - name: spot-workers
    instanceTypes: ["t3.medium", "t3.large"]
    spot: true
    minSize: 0
    maxSize: 5
    desiredCapacity: 2
    tags:
      Environment: production
      NodeType: spot

cloudWatch:
  clusterLogging:
    enableTypes: ["*"]
EOF

# Create cluster with config file
eksctl create cluster -f cluster-config.yaml

Method 2: Using AWS Console

  1. Navigate to EKS in AWS Console
  2. Click "Create cluster"
  3. Configure cluster name, version, and role
  4. Set up networking (VPC, subnets, security groups)
  5. Configure logging and add-ons
  6. Create worker node groups
  7. Configure kubectl access

Deploy with AWS Load Balancer Controller

# Install AWS Load Balancer Controller
curl -o iam_policy.json https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/v2.5.4/docs/install/iam_policy.json

# Create IAM policy
aws iam create-policy \
  --policy-name AWSLoadBalancerControllerIAMPolicy \
  --policy-document file://iam_policy.json

# Create service account
eksctl create iamserviceaccount \
  --cluster=my-cluster \
  --namespace=kube-system \
  --name=aws-load-balancer-controller \
  --role-name AmazonEKSLoadBalancerControllerRole \
  --attach-policy-arn=arn:aws:iam::ACCOUNT-ID:policy/AWSLoadBalancerControllerIAMPolicy \
  --approve

# Install the controller using Helm
helm repo add eks https://aws.github.io/eks-charts
helm repo update

helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
  -n kube-system \
  --set clusterName=my-cluster \
  --set serviceAccount.create=false \
  --set serviceAccount.name=aws-load-balancer-controller

Deploy with Ingress

# Create deployment and service
cat << EOF | kubectl apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: web-app-service
spec:
  selector:
    app: web-app
  ports:
  - port: 80
    targetPort: 80
  type: ClusterIP
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: web-app-ingress
  annotations:
    kubernetes.io/ingress.class: alb
    alb.ingress.kubernetes.io/scheme: internet-facing
    alb.ingress.kubernetes.io/target-type: ip
spec:
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: web-app-service
            port:
              number: 80
EOF

# Check the ingress
kubectl get ingress
kubectl describe ingress web-app-ingress

Using ECR for Container Images

# Create ECR repository
aws ecr create-repository --repository-name my-app --region us-west-2

# Get login token
aws ecr get-login-password --region us-west-2 | docker login --username AWS --password-stdin ACCOUNT-ID.dkr.ecr.us-west-2.amazonaws.com

# Tag and push your image
docker tag my-app:latest ACCOUNT-ID.dkr.ecr.us-west-2.amazonaws.com/my-app:latest
docker push ACCOUNT-ID.dkr.ecr.us-west-2.amazonaws.com/my-app:latest

# Deploy from ECR
kubectl create deployment my-app --image=ACCOUNT-ID.dkr.ecr.us-west-2.amazonaws.com/my-app:latest

Auto Scaling

# Install Cluster Autoscaler
curl -o cluster-autoscaler-autodiscover.yaml https://raw.githubusercontent.com/kubernetes/autoscaler/master/cluster-autoscaler/cloudprovider/aws/examples/cluster-autoscaler-autodiscover.yaml

# Edit the file to add your cluster name and apply
kubectl apply -f cluster-autoscaler-autodiscover.yaml

# Annotate the deployment to prevent eviction
kubectl -n kube-system annotate deployment.apps/cluster-autoscaler cluster-autoscaler.kubernetes.io/safe-to-evict="false"

# Install Horizontal Pod Autoscaler
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml

# Create HPA for your deployment
kubectl autoscale deployment nginx --cpu-percent=50 --min=1 --max=10

# Check HPA status
kubectl get hpa

Security Best Practices

Essential Security Measures:
  • RBAC: Configure Role-Based Access Control
  • Network Policies: Restrict pod-to-pod communication
  • Pod Security Standards: Enforce security policies
  • Secrets Management: Use AWS Secrets Manager or Parameter Store
  • Image Scanning: Enable ECR image scanning
  • Private Endpoints: Use VPC endpoints for API access
# Enable private endpoint access
aws eks update-cluster-config \
  --name my-cluster \
  --resources-vpc-config endpointPrivateAccess=true,endpointPublicAccess=false

# Create RBAC for users
kubectl create serviceaccount my-user
kubectl create clusterrolebinding my-user-binding --clusterrole=view --serviceaccount=default:my-user

Backup and Disaster Recovery

# Install Velero for backup
wget https://github.com/vmware-tanzu/velero/releases/latest/download/velero-linux-amd64.tar.gz
tar -xzf velero-linux-amd64.tar.gz
sudo mv velero-*/velero /usr/local/bin/

# Configure S3 bucket for backups
aws s3 mb s3://my-cluster-backups-$(date +%s)

# Install Velero on cluster
velero install \
  --provider aws \
  --plugins velero/velero-plugin-for-aws:v1.7.0 \
  --bucket my-cluster-backups \
  --secret-file ./credentials-velero \
  --backup-location-config region=us-west-2

# Create backup
velero backup create my-backup --include-namespaces default
velero backup get

Cluster Maintenance

# Update cluster version
eksctl update cluster --name my-cluster --version 1.29

# Update node groups
eksctl update nodegroup --cluster my-cluster --name standard-workers

# Drain nodes for maintenance
kubectl drain NODE_NAME --ignore-daemonsets --force

# Uncordon nodes after maintenance
kubectl uncordon NODE_NAME

# Clean up cluster
eksctl delete cluster --name my-cluster
Cleanup Warning: Always delete your EKS cluster when done testing to avoid ongoing charges. Use `eksctl delete cluster --name my-cluster` to remove all resources.
